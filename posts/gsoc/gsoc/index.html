Introduction to SBOL Link to heading

My prior experience with the SBOL data format was visiting the SynBioHub website previously, as I understood it as a valuable location to hold biological data. I didn’t imagine that I would start working with this data format until the program started.

The aim of this project was to create a pipeline that would go from data represented in SBOL format to preprocessed data formats (vectors) that could be used for machine learning training. SynBioHub is the database with the largest amount of standardized biological designs encoded in SBOL and therefore is a great place to obtain a large amount of training data for ML models.

Classical ML models can be trained on experimental values such as:

  • RNA expression values
  • Encoded DNA sequences

These can be easily encoded in SBOL and uploaded to SynBioHub. However, because the data in SynBioHub is encoded in a triple-store XML format, these data values cannot be directly passed into ML models, which expect numeric features. Thus, if biological researchers wish to use the data in SynBioHub, they face the task of having to encode this data, which becomes infeasible without some experience in ML. As a result, SynBioHub currently lacks accessible, preprocessed datasets, which limits researchers’ ability to efficiently develop models trained on the data from SynBioHub.

Task One Link to heading

My first task was to encode data from a research paper into SBOL format. The context behind this paper is that these researchers wanted to explore how they can discover promoter sequences within the E. coli genome. During that process, they performed a lot of experiments and uploaded that data to GitHub.

With this data, they trained multiple machine learning models, attempting to:

  • Predict whether a given promoter sequence was inactive or active
  • Directly predict the expression level of the promoter

Because we wanted to create an SBOL to machine learning training pipeline, we decided to use their data to test out our package. Thus, my goal was to first encode this data from the paper into SBOL.

Tools and Documentation Link to heading

Because of my unfamiliarity with the SBOL data format, I read through a lot of documentation such as the SBOL specification, which describes the underlying structure, logic, and reasoning behind the format, and pySBOL2, which is a Python package that provides neat functions that abstract a lot of the SBOL logic, allowing the user to create SBOL files easily.

I spent a lot of time in this task rebuilding the SBOL format. The data from the paper could be encoded in many different ways, so we had to discuss and determine what was the best representation. Because I was working with rows of experimental data, my code for this section was essentially a single loop that created all the components and relationships needed to encode the data, which we iteratively improved over time.

At first, I was creating one SBOL file that encoded 17,000 rows of data, but later on we realized that it would make more sense to encode each row as an SBOL file and store the aggregate in a collection on SynBioHub.

Challenges Link to heading

We also struggled with encoding some important information in our representation, such as the RNA expression level, which is typically the target variable in synthetic biology ML. Initially we used SBOL attachments to encode the data, but eventually figured out how to use the SBOL Measurement class instead. This paid off in the end, as it allowed us to easily pull this information using SPARQL queries in our package.

Results Link to heading

By the end of this process, I was able to generate the SBOL files of the research data encoded in our desired representation. One key bottleneck that we noticed was that encoding each of the data points took a fair amount of time, as I had to leave my computer running for about a day.

The next step was to upload this data to SynBioHub. Luckily enough, there was a nicely documented API which used its endpoints to upload the data to SynBioHub automatically. Once the data was stored in a collection in SynBioHub, we also had to test how we could pull this data, which is also nicely implemented in the API.

The reason we uploaded the data to SynBioHub was to re-enact the general workflow that a user of our package would follow. Presumably, their data would already be stored within an SBOL collection, and the user would use the SynBioHub API to pull those files into their Python environment. Once those files are on the user’s computer, the package would be ready to be used.
